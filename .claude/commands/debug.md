다음 버그/에러를 MIT 체계적 디버깅 방법론(재현 → 원인 파악 → 수정)으로 분석합니다.

**입력된 버그**: $ARGUMENTS

---

## 🔍 Phase 1: 버그 재현 (Reproduce)

아래 단계로 버그를 재현하고 맥락을 수집합니다.

1. `$ARGUMENTS`에서 에러 메시지, 파일 경로, 라인 번호 파싱
2. 언급된 파일이 있으면 해당 파일과 주변 코드 읽기 (Read 도구)
3. 관련 파일 탐색 (Glob, Grep 도구)
4. 최근 git 변경사항 확인:
   - `git log --oneline -5`
   - `git diff HEAD --name-only`
5. 재현 가능한 최소 케이스 도출

재현이 불가능하거나 정보가 부족하면 사용자에게 추가 정보를 요청합니다.

출력 형식:
```
## 🔍 Phase 1: 버그 재현

**증상**: <에러 메시지 또는 관찰된 동작>
**관련 파일**: <파일 경로 목록>
**최근 변경사항**: <최근 커밋/변경된 파일>
**최소 재현 케이스**: <재현을 위한 최소 조건>
```

---

## 🧠 Phase 2: 원인 가설 (Locate — Hypothesis-Driven)

> **핵심 원칙**: 수정을 시도하기 전에 반드시 근본 원인을 먼저 찾습니다.
> 이 단계에서는 코드를 수정하지 않고 분석만 합니다.

Phase 1에서 수집한 데이터를 바탕으로:

1. 가설 3개 이상을 가능성 높은 순서로 나열
2. 각 가설의 근거 설명 (코드 증거 포함)
3. 각 가설을 검증/반증할 수 있는 실험 방법 제시 (수정 시도 금지)
4. 가장 유력한 원인 1개 지목 + 근거

출력 형식:
```
## 🧠 Phase 2: 원인 가설 (Hypothesis-Driven)

> 원칙: 수정 전에 반드시 근본 원인을 먼저 찾습니다.

| 순위 | 가설 | 근거 | 검증 방법 |
|------|------|------|-----------|
| 1 (유력) | ... | ... | ... |
| 2 | ... | ... | ... |
| 3 | ... | ... | ... |

**결론**: 가장 유력한 원인은 [가설 1]입니다. 이유: ...
```

---

## 🛠 Phase 3: 수정 계획 (Fix)

Phase 2의 결론을 바탕으로 수정 계획을 수립합니다.

1. 수정 필요 파일 목록 정리
2. 구체적 변경 사항 명시 (코딩 오류 vs 설계 오류 구분)
3. 유사 버그가 다른 곳에도 있는지 Grep으로 스캔
4. 사이드 이펙트 경고
5. **사용자 승인 요청** — 승인 후에만 실제 수정 진행
6. 수정 완료 후 회귀 테스트 방법 제안

출력 형식:
```
## 🛠 Phase 3: 수정 계획

**유형**: 코딩 오류 / 설계 오류
**수정 파일**: <파일 경로>
**변경 내용**:
  - <구체적 변경 내용 1>
  - <구체적 변경 내용 2>
**유사 버그 여부**: <다른 파일에서 동일 패턴 발견 여부>
**⚠️ 사이드 이펙트 주의**: <예상되는 부작용>

→ 위 계획대로 수정을 진행할까요?
```

승인을 받은 후에만 파일 수정을 진행하고, 완료 후 회귀 테스트 방법을 안내합니다.
